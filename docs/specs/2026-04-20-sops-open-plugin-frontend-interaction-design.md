# 标准运维开放插件接入 BKFlow 前端交互设计

## 背景与目标

标准运维开放插件接入 BKFlow 后，前端需要承接三类新心智：

- 空间级开放治理：某个空间当前能不能用某个开放插件
- 模板级版本绑定：模板节点最终绑定的是哪个 `plugin_version`
- 任务级异常解释：插件不可用类问题统一在任务页解释，而不是只在启动前拦截

这份设计的目标不是定义前端实现细节，而是为后续页面实现提供一组稳定的交互基线，使产品、后端和前端能围绕同一套页面结构、状态命名和提示语义协作。

## 交付范围

本次交付采用“轻量交付包”：

- 一条主流程原型
- 三组独立页面原型
- 一份前端交互说明文档
- 一份前后端协议文档

交付同时覆盖两类范围：

- `MVP`：能够支撑前端做页面设计和基础实现拆分
- 后续增强态：只记录关键预留，不在当前阶段展开成复杂产品方案

本次页面范围只覆盖：

- 空间开放插件管理页
- 模板编辑中的插件选择与版本展示页
- 任务页中的插件异常提示页

## 主流程说明

端到端主流程分为三步：

1. 空间管理员进入“开放插件管理”，查看来源目录、插件开启状态和同步结果，决定当前空间可使用哪些开放插件。
2. 模板作者在节点配置里按普通 API 插件方式选择来源、分类和插件，并显式选择或确认 `plugin_version`。
3. 用户启动任务后统一进入任务页。如果插件未开放、版本失效或来源不可达，任务页以结构化错误卡片解释问题，并引导用户回到模板页或空间治理页处理。

这里的设计重点是：

- 不新增一种插件类型心智
- 把 `plugin_version` 变成前端显式概念
- 把异常解释统一沉淀到任务页

## 页面关键信息结构

### 1. 空间开放插件管理页

页面需要同时表达四层信息：

- 当前空间接入了哪些开放插件来源
- 来源是否可用、最近同步时间是什么
- 来源下有哪些插件、插件当前是否已开启
- 管理员可执行哪些动作：单个开关、当前可见插件一键全开、来源级紧急停用（增强态）

推荐信息结构：

- 页面标题与说明
- 来源摘要卡片
- 搜索与筛选工具栏
- 插件表格
- 页面级规则说明与异常提示

### 2. 模板编辑插件选择与版本展示页

页面需要表达五层信息：

- 当前节点仍走普通 API 插件选择体验
- 插件来自哪个来源和哪种类型
- 当前绑定的是哪个 `plugin_version`
- 当前版本是否仍可继续新用
- 历史模板回看时，什么是只读快照，什么需要用户切版本

推荐信息结构：

- 插件选择区
- 插件摘要区
- 版本区
- 参数配置区
- 历史模板回看说明区

### 3. 任务页异常提示

页面需要表达四层信息：

- 任务为什么失败或为什么无法继续执行
- 这是治理问题、版本问题还是来源问题
- 当前任务受影响的范围
- 用户下一步应该去哪里处理

推荐信息结构：

- 任务基础信息
- 结构化错误卡片
- 建议动作区
- 历史快照说明区
- 增强态异常分类区

## 页面状态覆盖

### 空间开放插件管理页

至少覆盖以下状态：

- 初始空态：来源已接入但当前无目录
- 默认态：插件已同步且默认关闭
- 部分开启态：部分插件已开启
- 来源异常态：来源不可达或同步失败
- 插件下线态：插件保留记录但不可继续新用
- 增强态：来源级紧急停用、新增插件同步后默认关闭

### 模板编辑插件选择与版本展示页

至少覆盖以下状态：

- 正常可配置态
- 单版本态
- 多版本态
- 当前版本不可用态
- 当前空间已关闭该插件态
- 历史模板回看态
- 增强态：版本切换提醒、差异说明入口

### 任务页异常提示

至少覆盖以下状态：

- 插件未开放
- 插件版本不可用
- 插件已从目录下线
- 来源当前不可达
- 历史任务只读解释态
- 增强态：更细粒度错误来源分类

### 主流程原型

主流程原型至少要同时展示：

- 一条正常链路
- 一条版本失效异常链路

它不承担所有细枝末节，而是负责讲清楚“治理 → 配置 → 任务结果”的闭环。

## 关键交互说明

本次交付确认的关键交互如下：

1. 一键全开只作用于当前已发现插件  
后续来源新同步进来的插件仍默认关闭，不继承历史批量动作。

2. `plugin_version` 为显式字段  
模板编辑页正常态也直接显示 `plugin_version`，不把版本隐藏到摘要信息里。

3. 失效版本可回看、不可继续新用  
历史模板仍可查看旧版本快照，但不能直接基于失效版本继续创建或启动新任务。

4. 任务页负责解释，不负责原地修复  
当任务遇到插件未开放、版本失效或来源异常时，页面只给出原因和建议动作，不在任务页内直接修改配置。

5. 页面保持现有 BKFlow 心智  
来源变化和版本治理不改变“这仍然是普通 API 插件配置”的整体认知。

## 后续增强态预留

当前先记录以下增强方向，但不在本轮展开成复杂实现：

- 来源级紧急停用的完整交互
- 模板页的版本升级提示与字段差异入口
- 任务页中更细粒度的错误来源与跳转链接
- 按来源或按版本的更细粒度展示筛选

增强态的原则是：只增强信息密度和解释能力，不重写当前页面主结构。

## 非目标

本设计明确不包含：

- 前端业务代码实现
- 组件拆分与工程结构设计
- 详细测试用例矩阵
- 完整版本迁移器或差异比对器
- 重新定义 BKFlow 主 spec 的长期架构边界
