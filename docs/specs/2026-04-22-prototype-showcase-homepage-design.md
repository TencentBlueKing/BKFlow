# Prototype Showcase Homepage Design

> 日期：2026-04-22
> 状态：Draft（方案已确认，待书面审阅）
> 目标：将 `prototypes/` 根首页从“URL 路径列表”升级为面向评审与浏览的中文原型展厅首页

## 1. 背景

当前 `prototypes/serve.py` 在根路径 `/` 返回的是一个按 `masters / features / examples` 分组的 HTML 路径列表。它已经满足了“能打开页面”的最低要求，但在原型工具已经完成目录重构之后，这种入口暴露出三个明显问题：

1. **目录感太强**
   首页首先呈现的是文件路径，而不是原型本身的价值、范围和页面特征。对评审、产品、设计或第一次进入工具的人来说，理解成本偏高。
2. **与 feature 目录模型不匹配**
   新版目录已经明确采用 `masters/` 作为长期母版库、`features/<slug>/` 作为需求级交付目录，但首页没有把 `feature` 作为主入口来表达，仍然停留在“静态文件浏览器”层面。
3. **观感落差明显**
   `features/bkflow-engine-admin-prototype-overhaul/index.html` 已经具备较完整的中文标题、卡片分组和展厅式观感，而工具首页仍是默认列表风格，造成入口质量和 feature 内页质量不一致。

用户期望首页更接近 Figma 社区、Codesign 一类“可浏览的作品展厅”，但仍要保留 BKFlow 原型工具的可维护性和目录治理能力。

## 2. 设计目标

本次首页改造的目标如下：

1. **首页先表达作品，而不是路径**
   用户进入后先看到“有哪些原型交付”，再决定进入哪一组页面。
2. **以 `features/` 为首页主角**
   首页优先服务成套需求原型浏览，`masters` 与 `examples` 保留但弱化。
3. **提供中文化展示信息**
   首页应展示中文标题、中文摘要、中文标签与中文状态，不能要求用户通过 slug 或路径推断含义。
4. **保留轻量维护成本**
   不引入真实截图流水线；首页使用代码生成的封面卡片和轻量元数据即可接入新 feature。
5. **继续兼容当前静态工具链**
   保持 `serve.py + HTML/CSS/JS` 的零 Node、零构建模式，不引入新的运行时依赖。

## 3. 非目标

本次设计明确不做以下事情：

1. 不把首页改造成完整的原型管理后台。
2. 不引入真实页面截图生成、存储、同步机制。
3. 不改变 `features/<slug>/index.html` 作为 feature 内部入口页的职责。
4. 不要求 `masters/`、`examples/` 也拥有与 `features/` 同等级的首页视觉权重。
5. 不引入数据库、服务端模板引擎或前端框架。

## 4. 方案选型

围绕首页的表达方式，评估三种方向：

| 方案 | 思路 | 优点 | 缺点 |
|------|------|------|------|
| **A. Feature 展厅首页** | 首页以 `feature` 大卡片为主，下方辅以代表页面与资源区 | 最符合“作品展厅”目标，同时保持目录治理关系 | 需要补充 feature 元数据契约 |
| B. 纯页面画廊 | 首页直接铺开所有具体页面卡片 | 观感最强，像设计稿画廊 | 页面一多就会失控，弱化 feature 组织 |
| C. 工作台式首页 | Feature、masters、examples 平级展示 | 信息完整 | 仍然偏内部工具，不像展厅 |

**最终选择方案 A。**

选择理由：

1. 用户明确希望首页更像“展厅”，而不是“目录页”。
2. `features/` 已经是新版原型组织的主轴，首页应强化而不是绕开这条主线。
3. 方案 A 能兼顾观感与可维护性，不会因为追求画廊效果而打散 `spec / plan / feature` 的绑定关系。

## 5. 总体设计

首页保持由 `prototypes/serve.py` 在 `/` 动态生成，但生成结果不再是简单列表，而是一个中文化的原型展厅首页。整体信息架构分为 4 层：

### 5.1 Hero 首屏

首页顶部为展厅首屏，职责是建立第一印象和主导航意图。

应包含：

- 中文主标题，例如“BKFlow 原型展厅”
- 一句中文副标题，说明这是“按需求组织的高保真交互原型入口”
- 3 组以内的关键统计信息，例如：
  - Feature 数量
  - 页面总数
  - 母版数量
- 一个主操作按钮，例如“浏览最新原型”
- 一个次级按钮或锚点，例如“查看工具资源”

Hero 的角色是“告诉用户这是什么”，而不是“立即列路径”。

### 5.2 Feature 展厅主区域

首页主体是 `features/` 的大卡片展区，每张卡片代表一个 `features/<slug>/` 目录。

每张卡片必须包含：

- 中文标题
- 中文摘要
- 状态标签
- 页面数量
- 引用母版数量（如果有）
- 设计化封面
- 主按钮“查看这组原型”

卡片整体可点击进入 feature 入口页，但仍保留按钮以降低理解成本。

### 5.3 代表页面预览区

在 feature 主区域之后，展示跨 feature 挑选出来的代表页面预览卡片。其目的不是替代 feature 内页，而是让用户快速建立“这套原型大概长什么样”的感知。

每张预览卡片应展示：

- 页面中文名
- 一句话摘要
- 页面类型标签，如“流程编辑”“任务详情”“列表页”
- 所属 feature
- 对应入口链接

缩略图不使用真实截图，而是采用代码生成的抽象页面封面。

### 5.4 工具资源次级区

`masters/`、`examples/`、`README` 等工具资源放在首页底部次级区域，以轻量卡片或简短入口列表呈现。

该区域职责是：

- 为维护者保留入口
- 弱化目录信息对首页主视觉的干扰
- 避免 `masters / examples` 与 `features` 在首页争夺同一层级

## 6. Feature 卡片设计

Feature 卡片采用 `封面 + 信息 + 行动` 三层结构。

### 6.1 封面层

封面不是页面截图，而是代码生成的设计化主题卡面。封面用于表达这组原型的大致气质，而不是精确复刻某个页面。

首批支持的封面主题建议包括：

- `flow-editor`
- `task-detail`
- `list-management`
- `config-panel`
- `mixed-admin`

封面可由几种稳定的抽象元素组合而成，例如：

- 顶部操作条
- 左侧列表栏
- 中央画布/表格区
- 右侧抽屉
- 状态徽标或节点块

这样可以保证：

- 没有截图维护成本
- 风格统一
- 新 feature 只需声明主题即可接入

### 6.2 信息层

Feature 卡片的信息层全部以中文展示为主，slug 只在次要位置存在，或不在首页直接展示。

推荐字段：

- `title`：中文标题
- `summary`：一句话说明覆盖范围
- `status`：例如“可评审”“建设中”“已归档”
- `tags`：如“流程编辑”“任务详情”“系统配置”
- `pageCount`
- `masterCount`
- `updatedAt`

### 6.3 行动层

推荐为每张卡片提供：

- 主按钮：`查看这组原型`
- 次按钮：`查看代表页面`

卡片 hover 时可以增强按钮可见性，但不应隐藏所有主要行动，避免桌面端与移动端体验差异过大。

## 7. 数据来源设计

首页数据采用 **自动扫描 + 轻量元数据** 的双层模式。

### 7.1 自动扫描层

`serve.py` 继续负责扫描以下目录：

- `prototypes/features/`
- `prototypes/masters/`
- `prototypes/examples/`

自动收集的事实数据包括：

- feature 数量
- 页面数量
- 每个 feature 的 HTML 页面清单
- 最近修改时间
- `masters` 与 `examples` 的可访问入口

这层只负责客观事实，不承载人工展示文案。

### 7.2 轻量元数据层

每个 `features/<slug>/` 目录新增一个轻量元数据文件：

- `feature.meta.json`

该文件只承载首页展示需要的人类可读信息，不替代 `README.md`。

建议结构如下：

```json
{
  "title": "BKFlow 引擎管理后台原型重构",
  "summary": "覆盖 Space / System / Plugin 三个入口，包含流程、任务、决策与配置链路。",
  "status": "可评审",
  "tags": ["流程编辑", "任务详情", "系统配置"],
  "coverTheme": "mixed-admin",
  "order": 10,
  "featuredPages": [
    {
      "path": "pages/space/flow-edit.html",
      "title": "流程编辑页",
      "summary": "画布编排、节点抽屉与两段式保存。",
      "pageType": "流程编辑"
    },
    {
      "path": "pages/space/task-detail-complete.html",
      "title": "任务详情页",
      "summary": "执行状态、节点观察和日志查看。",
      "pageType": "任务详情"
    }
  ]
}
```

### 7.3 为什么不用硬解析 README

虽然 `README.md` 已经存在并包含 feature 描述，但首页不应依赖 Markdown 文本解析来获得展示信息，原因如下：

1. README 的职责偏向说明与追踪，不是稳定数据接口。
2. Markdown 结构容易在人工编辑时漂移。
3. 首页生成逻辑需要稳定、可测试、低歧义的字段源。

因此推荐把 `README.md` 视为文档，把 `feature.meta.json` 视为首页数据契约。

## 8. 降级与容错策略

首页需要对不完整数据保持可用，而不是在某个 feature 缺字段时整体失效。

### 8.1 缺少 `feature.meta.json`

如果某个 feature 目录没有元数据文件：

- 用 slug 生成一个临时标题
- `summary` 使用默认占位文案，如“尚未补充首页摘要”
- `status` 标记为“待补充”
- `coverTheme` 使用默认主题 `mixed-admin`
- `featuredPages` 默认选取前 2 个代表性 HTML 页面

### 8.2 元数据中的页面路径无效

如果 `featuredPages.path` 指向不存在的页面：

- 跳过该页面卡片
- 不影响首页其他区域渲染

### 8.3 无 feature 数据

如果 `features/` 下没有任何有效 feature：

- 首页仍显示 Hero 区
- 主区域显示友好的空状态，而不是空白页
- 工具资源区照常显示

### 8.4 `masters/examples` 缺少说明信息

首页底部资源区可以退化为固定入口卡片，不依赖复杂元数据。

## 9. 交互设计

首页交互应保持轻量，但要明显区别于纯静态目录页。

### 9.1 搜索

提供一个首页搜索框，支持按以下字段做前端过滤：

- feature 中文标题
- slug
- tags
- featured page 标题

搜索不需要服务端参与，只需在浏览器端根据已渲染的数据属性过滤卡片。

### 9.2 筛选

首页提供轻量筛选项，首版建议支持：

- 状态
- 页面类型标签

筛选控件应保持简洁，避免把首页做成复杂后台。

### 9.3 最近更新

首页 Hero 下方或 Feature 区顶部可增加“最近更新”提示，基于自动扫描得到的修改时间排序。

它的作用是帮助用户优先看到刚更新的原型，而不是成为一个独立页面列表。

### 9.4 悬浮反馈

Feature 卡片 hover 时建议提供：

- 卡片轻微抬升
- 边框或投影增强
- 封面元素轻微位移
- 行动按钮更明显

但动画需保持克制，不做营销化或过重的过场。

## 10. 视觉方向

首页整体风格采用 **“轻展厅 + 工具理性”** 的方向。

具体原则：

1. 背景不使用纯白文件页，而采用浅色渐变、轻网格或弱光斑。
2. Hero 区需要明显的品牌感，但不做营销站式夸张视觉。
3. Feature 卡片尺寸偏大、中文标题清晰、层级明显。
4. 路径与 slug 退到弱层级，只保留给维护者识别。
5. `masters / examples` 入口使用较轻的视觉权重，作为资源区存在。

## 11. 实现边界

虽然本文件不进入实现细节，但需要提前定义几个边界，避免后续实现偏航：

1. **首页继续由 `serve.py` 生成**
   不新建一个固定的 `index.html` 作为首页真源。
2. **允许引入新的共享静态资源**
   如有必要，可新增专门的首页 CSS / JS 资源文件放在 `prototypes/assets/` 下，用于承接展厅视觉和前端筛选逻辑。
3. **不引入构建链**
   所有视觉与交互仍保持在静态 HTML/CSS/JS 范围内。
4. **不依赖真实截图**
   封面一律由代码生成。

## 12. 测试与验收

首页改造完成后，至少需要覆盖以下验证点：

### 12.1 结构测试

- 根首页包含 Hero、Feature 区、代表页面区、资源区
- 首页不再输出简单的路径列表结构作为主布局
- `features` 成为首页主区域而非次级列表

### 12.2 元数据解析测试

- 能正确读取 `feature.meta.json`
- 缺少元数据时能回退到默认展示
- `featuredPages` 无效路径不会导致渲染失败

### 12.3 文案与统计测试

- 首页显示中文标题和中文摘要
- 页面数、feature 数等统计值与扫描结果一致
- 最近更新时间或最近更新排序逻辑正确

### 12.4 交互测试

- 搜索能过滤 feature 卡片
- 状态/标签筛选能联动隐藏和显示卡片
- 卡片链接能正确跳转到 feature 入口页或代表页

### 12.5 兼容性测试

- 桌面端下卡片布局和 Hero 区正常
- 移动端下卡片能单列阅读，不出现严重溢出

## 13. 成功标准

本次首页改造完成后，应达到以下结果：

1. 打开 `http://localhost:9080/` 时，用户第一眼看到的是中文原型展厅，而不是文件索引页。
2. 用户无需读 URL 路径，就能理解每组 feature 的用途和覆盖范围。
3. 新增 feature 只需补一个目录和一份轻量元数据，即可被首页自动纳入展厅。
4. `masters` 与 `examples` 仍然可访问，但不会抢占首页主视觉焦点。
5. 工具仍保持零构建、可本地运行、可测试的静态模式。
