# TAPD 提测单模板

本文档用于沉淀 BKFlow 项目通用的 TAPD 提测单结构，适用于「功能提测」「联调后转测」「版本验收前集中提测」场景。

## 使用说明

- 适用 TAPD 空间：`BKFlow`（`workspace_id=xxx`）
- 建议单据类型：`stories`
- 建议标题格式：`[提测] <模块/功能> <版本或里程碑>`
- 填写顺序：先补齐背景与版本说明，再补测试范围与验收标准，最后补测试执行信息

## 标准模板（可直接复制）

```markdown
### 一、提测背景
<说明此次提测的背景、目标和价值，1-3 段>

- PR/分支：<链接或标识>
- 关联单据：<story/bug id 列表>
- 需求来源：<产品需求/技术需求/缺陷修复>

### 二、版本说明
- [新增] ...
- [增强] ...
- [修复] ...
- [优化] ...

### 三、影响范围
- 后端：<涉及模块/接口/任务链路>
- 前端：<涉及页面/组件/交互>
- 数据/依赖：<依赖升级、配置调整、迁移说明>

### 四、测试范围（P0）
1）功能主链路
- ...

2）异常与边界场景
- ...

3）关键兼容性场景
- ...

### 五、回归范围（P1）
- ...

### 六、风险点与重点关注
- ...

### 七、验收标准
- P0 用例全部通过，无 blocker/critical 缺陷
- 回归通过，核心链路无明显性能退化
- 线上兼容性满足预期（如适用）

### 八、测试执行信息
- 提测环境：<环境名>
- 提测版本/镜像：<版本号>
- 测试负责人：<人员>
- 计划完成时间：<日期>
```

## TAPD 字段建议

- `priority_label`：默认 `High`（可按实际调整）
- `category_id`：默认建议 `新功能`，优化类改动可选 `优化需求`
- `iteration_id`：优先选当前开放迭代
- `owner`：默认填写测试负责人

## 质量建议

- 避免仅写「请测试」，应明确可验证的验收结果
- 每个 P0 场景尽量可映射到 1 组可执行用例
- 风险点优先写「最可能导致回归/事故」的 3-5 条
